# 엘리트 웹 UI 퀄리티 — 딥리서치 리포트

> 소스 30개 · 검증 통과 주장 58 · 과장으로 반박·제외 17. 아래는 검증 통과분만.
> (딥리서치 워크플로 wf_d7742539에서 복구·재합성)

## 0. 한 줄 결론 (냉정하게)

네가 본 "Fable 5로 몇 분 만에 $10k 시네마틱 3D 사이트" 영상의 실제 정체:
**Higgsfield는 코드·웹사이트·3D를 만들지 않아 — 이미지/영상 "에셋"만 생성하는 도구야.**
즉 그 결과물 = ① Higgsfield가 시네마틱 *소재*(이미지·15초 영상) 생성 → ② Claude/Fable이 *사이트를 코딩* → ③ 사람이 마감. "AI가 통째로 뽑았다"가 아니야.

그리고 더 중요한 것: **그건 랜딩페이지 기법이고, 우리 변환기는 기능형 툴이야.** 리서치가 반복해서 말하는 건 — 시네마틱 스크롤·3D는 **여가/쇼케이스용**이고 **작업 중심 툴에선 오히려 해가 돼**. 네가 과한 걸 바란 게 아니라, *랜딩 기법을 툴 화면에 겨눴던 것*뿐이야. 둘을 분리하면 다 풀려.

## 1. "고급"을 만드는 실제 기법 (검증됨)

**철학 — 가장 단순한 수단으로 최대 임팩트.** 최고 사례들의 공통 원칙은 "가장 복잡한 기술이 아니라 가장 강한 인상을, 가장 단순한 방법으로"였음. 깊이·라이팅·그레인을 *진짜 3D 대신 가짜로* 냄. **성능을 디자인 시스템의 일부로** 취급하고, 인터랙션은 장식이 아니라 *구조를 드러내려고* 씀. (Codrops: The Sleepers, Exat)

**모션의 '고급감'은 이징 곡선에서 나온다.** 프리미엄 스태거 리빌은 linear/기본 이징이 아니라 **겹겹의 파워 커브(power3.in 등)**를 씀. 이게 "폴리시된 느낌"의 정체. (Eloy Benoffi 수상작 스택)

**타이포그래피를 '재료'로.** 수상작들은 **가변 폰트 축(axis)을 인터랙션으로** 씀 — 호버하면 굵기/폭이 실시간 모핑, opsz 축으로 크기별 자형 자동 조정. 커서-글자 거리 기반 마이크로인터랙션("영향권"). "시네마틱 3D 타이포"조차 실제론 **WebGL이 아니라 CSS/GSAP의 X축 회전 트릭**인 경우가 많음. (Exat 마이크로사이트)

**필름 그레인·왜곡(그 '질감').** GLSL simplex 노이즈를 프래그먼트 셰이더에서 돌리고, **Lenis 스무스 스크롤의 속도값을 셰이더 uniform으로 먹여** 스크롤에 반응하는 왜곡을 만듦. 단, 노이즈 함수는 표준이 아니라 갈아끼우는 것(과장 반박됨). (Codrops 셰이더 튜토리얼)

**마이크로인터랙션 원칙.** "거의 안 보이고 완전히 기능적"이어야 함. 주 목적은 **시스템 상태 전달**(진행률 등). 앱 전체 스타일과 일관돼야 함. **반복되면 참신함은 짜증으로 바뀜** → 매일 쓰는 UI일수록 절제. (Smashing Magazine)

## 2. 도구 스코어카드 (역할·난이도·번들, 검증됨)

| 도구 | 역할 | 난이도/비용 | 판정 |
|---|---|---|---|
| **GSAP + ScrollTrigger** | 스크롤 연출·복잡 애니 | 2–4주 학습, ~23–30KB | 수상작에 반복 등장(Exat, Eloy Benoffi=Awwwards HM/GSAP SotD). 랜딩용 정석 |
| **Framer Motion** | React **상태 전환**(모달·탭·리스트) | ~34–46KB | **"사이트 아니라 앱"용** — 기능형 UI의 정답. AnimatePresence 퇴장 애니 |
| **Motion.dev** | 경량 대안 | ~8KB, GSAP보다 빠름(벤치) | 성능 우선이면 후보 |
| **Lenis** | 스무스 스크롤(+셰이더에 속도 전달) | 경량 | 랜딩 스크롤 필수템 |
| **Three.js** | 진짜 3D/WebGL | **2–4개월** 학습(가파름) | 랜딩 히어로 전용. 툴 UI엔 과함 |
| **R3F** | JSX로 3D | 코드 중심, 자동 최적화 없음 | 개발자용 |
| **Spline** | 노코드 3D | 런타임 성능 '중', 무거우면 로딩 지연 | 프로토타입/가벼운 씬만 |
| **가변 폰트** | 타이포 품질 | 단일 파일이 정적 합본보다 30–50%↓ | opsz 축으로 12px·36px 둘 다 예쁨. **강력 추천** |
| **node-vibrant** | 앨범아트 색 추출 | **20KB**, 6색 스와치+대비안전 텍스트색 | 브라우저에서 직접 실행 가능 → 확장에 최적 |
| **colorthief** | 색 추출(단순) | RGB 배열만, **1.48MB** | node-vibrant이 우세 |
| **CSS scroll-driven(animation-timeline)** | 스크롤 애니 | 선언형 | **메인스레드 밖에서 실행** → JS 스크롤 리스너보다 안 끊김 |

## 3. AI 디자인 도구의 실체 (냉정하게, 검증됨)

- **Higgsfield MCP** = **이미지/영상 에셋 생성기**(30+ 모델, 이미지 4K, 영상 ≤15초). **코드·웹·3D·UI 생성 아님.** 크레딧 과금(생성마다 비용). "3D 사이트를 만든다"는 건 오해 — 소재만 공급.
- **Lovable** = 프로덕션 준비도 **60–70%**. 나머지 '마지막 30%'(복잡 로직·성능·보안·엣지케이스)는 수작업 필수. 프로토타입·랜딩·내부툴에 적합, 복잡 프로덕션엔 부적합.
- **v0 vs Lovable 미감 차이**: v0는 차분·진중, Lovable은 "반짝이·장식" 경향. **즉 미감은 도구+프롬프트에 좌우되지 자동이 아님.** AI 도구는 깨진 UI(흰 글자on흰 배경, 중복 아이콘)도 실제로 뱉음. Bolt는 버그 많음.
- **결론**: "몇 분 $10k"는 *진짜 에셋 + 진짜 코딩툴*의 조합이지만, **완성 마감은 사람 몫**. Magic Patterns가 우리한테 뽑아준 것도 "좋은 출발점"이지 완제품 아님(그래서 네가 부족함을 느낀 것 = 정상적 안목).

## 4. 모션이 도움 vs 해가 되는 경계 + 접근성 (검증됨)

- **스크롤 애니가 텍스트 표시를 지연시키면 작업형 사용자는 좌절.** 스크롤 연출은 여가/엔터용으로 아껴 쓰고, **작업 중심 플랫폼(금융·의료·B2B·툴)에선 피하라.** 본문 텍스트엔 X, 보조 그래픽에만. 첫 스크롤에만, 반복 재생 금지. (NN/g)
- **접근성(우리도 지켜야 함)**: `prefers-reduced-motion`(2020.1부터 전 브라우저 지원)으로 장식 모션 차단. 5초 넘는 자동 애니는 정지 컨트롤 필요(WCAG 2.2.2). 초당 3회 초과 점멸 금지. **성인 35%가 40세까지 전정기능 이상** → 큰 객체의 확대/패닝(시네마틱 패럴랙스의 핵심)이 어지럼 유발 → **모션 토글 기본 OFF**.
- **→ 우리 확장은 '툴'이라, 팝업·탭에 시네마틱 스크롤/3D는 넣으면 안 됨.** 모션은 절제·상태전달·reduced-motion 존중으로.

## 5. 크롬 확장 MV3 하드 제약 (검증됨)

- 기본 CSP: `script-src 'self'; object-src 'self'` → **인라인 스크립트·핸들러 금지**, 모든 JS는 외부 파일.
- **원격/CDN 코드 실행 불가** → GSAP·Three.js·Lenis·Lottie는 **로컬 번들 필수**. (폰트는 예외 — 원격 폰트는 가능하나, 안정성 위해 번들 권장)
- `unsafe-eval` 추가 불가, `eval`/`new Function` 금지(샌드박스 iframe 예외).
- **WASM 기본 비활성** → CSP에 `wasm-unsafe-eval` 넣어야 동작(일부 무거운 그래픽 라이브러리가 wasm 씀 → 주의).
- **팝업 최대 800×600px** → 더 큰/풀블리드 화면은 전용 탭/윈도우로. (우리 구조 이미 맞음)

## 6. 우리 프로젝트 액션 플랜 (핵심)

### A. 확장 UI (팝업 380px + 탭) — "절제된 에이전시급"
리서치 결론상 **여기엔 3D·시네마틱 스크롤을 넣지 않는다.** 대신 아래로 격을 올림:

1. **앨범아트 동적 테마** ← 지금 당장 가장 큰 한 방. `node-vibrant`(20KB, 로컬 번들)로 재생목록/재생 중 곡의 색을 추출 → **배경·글로우 같은 장식에만** 적용(=YouTube/Spotify 패턴), 텍스트·버튼은 고정 고대비 유지(WCAG 4.5:1 체크). 진짜 Spotify 느낌은 여기서 나옴.
2. **가변 폰트 로컬 번들** — opsz 축 있는 세리프 디스플레이 + 가변 본문. 12px 목록과 큰 헤드라인이 한 파일로 둘 다 예뻐짐. 지금 세리프 방향 유지.
3. **모션 = Framer Motion(또는 GSAP/Motion.dev) 로컬 번들** — 상태 전환(목록 스태거 등장, 검토→해결, 진행률)에 **파워 커브 이징**으로 절제 있게. `prefers-reduced-motion` 존중, 토글 제공.
4. **마이크로인터랙션** — 상태 전달형(변환 진행·매칭 확정 피드백), "거의 안 보이게". 목록 스태거는 GSAP이 React state보다 부드러움(60fps).
5. **질감** — 무거운 셰이더 대신 **정적 노이즈 텍스처 오버레이**(가벼움)로 따뜻한 그레인. '가장 단순한 수단' 원칙.

### B. 홍보 랜딩페이지 (선택, 별도 프로젝트) — "풀 시네마틱"
여기서 그 영상의 감성을 마음껏:
- GSAP + ScrollTrigger + SplitText + Lenis로 스크롤 연출. 히어로는 Three.js/R3F 또는 **2D 트랜스폼 트릭**(더 싸고 충분). 필름 그레인 셰이더.
- **Higgsfield로 히어로 영상/이미지 '에셋' 생성** → 사이트는 Claude/Fable/v0로 코딩 → **마지막 30% 수작업 마감.**
- 진짜 3D면 KTX2 텍스처·인스턴싱으로 성능. reduced-motion 존중.
- 이건 확장이랑 **완전 분리** (확장 스토어 등록물이 아니라 마케팅 사이트).

### 추천 순서
① 앨범아트 색 테마 + 가변폰트(확장에 바로, 체감 큼) → ② 상태 전환 모션 다듬기 → ③ (원하면) 랜딩페이지 별도 제작.
그리고 네가 넣은 **Apple Music 토글**은 A단계 색/컴포넌트 시스템 위에 소스만 추가하면 되게 설계할게.
